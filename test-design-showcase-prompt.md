# 指示: test-design スキルのショーケース OSS を作る

`test-design` スキル(`~/.claude/skills/test-design/`)が持つテスト設計手法を、実際に動く OSS として
「全手法が一直線に辿れる」形で実演するリポジトリを作る。スキルのワークフロー(0段の振る舞い抽出 →
手法割当 → 検証三層への橋渡し)を、コードと README で実地になぞれる教科書的ショーケースにする。

## 進め方の大方針

- **このスキルを必ず使う**。`test-design` スキルを起動し、その reference を実際に読みながら設計する。
  README とテストの構成は、スキルが定義する手法・網羅基準・落とし穴に1対1で対応させる。
- t-wada スタイルの TDD で駆動する(テストリスト先行 → Red → Green → Refactor)。
- まず **duration 題材を1本、縦に貫いて完成**させる(0段からテストまで)。残り題材は横に足せる骨組みだけ。
- 言語は TypeScript、テストランナーは vitest、property-based は fast-check
  (test-design の reference の example と一致させ、example がそのまま動くコードになるようにする)。
- 過剰に作らない(YAGNI)。各題材は数十行で済む最小実装。テストのために肥らせない。

## リポジトリ構成(目標)

```
test-design-showcase/
├─ README.md            … スキルのワークフローの説明と、各題材がどの手法を実演するかの対応表
├─ TESTPLAN.md          … 各題材の振る舞い抽出台帳(T-ID)。0段(用語定義→抽出→種別)を実物で見せる
├─ package.json         … vitest + fast-check
├─ src/
│  ├─ duration/         … parse ⇔ format(最初に縦に完成させる題材)
│  ├─ order-fsm/        … 注文の状態機械(状態遷移・無効遷移・MBT)
│  ├─ expr/             … 四則+変数の式評価器(構造網羅・メタモルフィック)
│  └─ llm-validator/    … 非決定出力の階層化バリデータ(入口/中央/出口)
└─ test/                … 各 src に対応するテスト(手法ごとにファイルやdescribeを分ける)
```

## 題材ごとに実演する手法(test-design の reference に対応づける)

| 題材 | 実演する手法 | 対応 reference |
|---|---|---|
| **duration**(`"1h30m"` ⇄ 秒) | 同値分割、境界値、PBT(round-trip: format∘parse=id)、差分テスト(参照実装と)、形式検証連携 | blackbox-systematic, modern-generative |
| **order-fsm**(created→paid→shipped→delivered) | 状態遷移テスト(0-switch 全遷移+無効遷移)、モデルベーステスト、デシジョンテーブル | blackbox-systematic, modern-generative, (loop-engineering 連携) |
| **expr**(`1+2*x` 評価) | C0/C1/MC/DC、デシジョンテーブル、メタモルフィック(可換・結合)、ループ/境界 | whitebox-coverage, blackbox-systematic, modern-generative |
| **llm-validator**(出力をスキーマ強制+合意+判定外部化) | 非決定的出力の階層化品質設計(入口層=入力検証を網羅、中央層=性質と複数サンプリング合意、出口層=後段バリデーション) | ai-nondeterministic |

## duration 題材の作り方(これを最初に縦に完成させる)

test-design スキルのワークフローを順に踏む。各ステップの成果を TESTPLAN.md に残す。

### 0段: 振る舞いを抽出する(いきなりテストを書かない)
1. **用語定義を先に固める**(スキルの 0段の最初のステップ)。「duration とは何か」「受理する単位は
   `h`/`m`/`s` か、`d` も含むか」「順序の制約はあるか(`30m1h` を許すか)」「ゼロ・負・空・オーバーフローの扱い」を
   曖昧さが残らないところまで言語化する。語の未定義が境界・異常系の抽出漏れ源になる。
2. その定義から、テストすべき振る舞いを `T-001` から採番で TESTPLAN.md に列挙する。
   正常系・境界・異常系・(必要なら)非機能を、過剰抽出は安全・漏れは危険の原則で出し切る。
3. 各 T-ID に種別を振り、各正常系に「不正入力なら? 境界なら?」を問うて異常系を系統的に生やす。

### 手法割当 → テスト(Red→Green を回す)
- **同値分割**: 入力文字列を有効クラス(単一単位、複合単位)と無効クラス(空、未知単位、非数値)に割り、各代表を `it.each` で。
- **境界値**: `0s`、最大値、桁あふれ、各単位の繰り上がり境界。
- **PBT(round-trip)**: 任意の非負秒数 `n` について `parse(format(n)) === n`(format∘parse=id)を fast-check で。
  生成器は広く取る(狭いと境界を踏まない)。
- **差分テスト**: 素朴な参照実装(正規表現ベース等)を別に書き、任意入力で本実装と一致を fast-check で突き合わせる。
  両実装が同じ穴を共有しないよう独立に書く。
- **形式検証連携(裏方、任意)**: `format∘parse=id`(または `parse∘format=id`)を Lean 4 で証明し、
  証明済み述語を上の PBT の property としてそのまま叩く。
  **証明物(.lean/lakefile)は `tasks/fv/` 配下で作り、リポジトリの git 管理には含めない**。
  README やコミットに形式検証の存在を書かない。結果(保証された往復性質と、それを叩く property)だけを反映する。

### 検証三層の使い分け(スキルの役割分担に従う)
- 実装の駆動: TDD(このリポジトリで回す)。
- 設計に状態遷移があるとき(order-fsm): 全 interleaving はテストで踏めない。必要なら loop-engineering(TLA+)で
  設計をモデル検査し、反例を Gherkin の受け入れシナリオに落としてからテストへ移す。
  **その中間生成物(.tla/.cfg/.feature/管理番号)は `tasks/loopeng/` 配下に置き git 管理に含めない**。
  本体ソース・テスト名・コミットに管理番号や手法の用語(S-001/R-xx 等)を漏らさない。
- critical なアルゴリズム性質: 通常テストより強い保証が要るなら Lean で証明し、証明済み述語を property として叩く。

## README.md に書くこと
- このリポジトリは何か(test-design のワークフローを実地でなぞるショーケース)。
- スキルのワークフロー(0段 → 手法割当 → 検証三層)の要約と、上の題材×手法の対応表。
- 各題材ディレクトリの README で「どの振る舞いに、なぜその手法を割り当てたか」を1対1で説明する。
- 形式検証・TLA+ の存在は書かない(裏方ルールに従う。結果の性質だけを自然な言葉で述べる)。

## 完成の定義(duration 題材)
- TESTPLAN.md に用語定義と T-ID 台帳があり、全 T-ID にテストが対応している(トレーサビリティが追える)。
- 同値分割・境界値・PBT・差分の4手法すべてのテストが緑。
- `npm test` が通る。README に手法対応表がある。
- 残り3題材(order-fsm/expr/llm-validator)はディレクトリと README の見出し、テストリストの骨組みだけ用意し、
  「次にここを縦に貫く」と分かる状態にする(全部は実装しなくてよい)。

## やらないこと
- 自明な1行や設定値にテストを先行させない(YAGNI)。
- カバレッジ率を目的化しない。テストの強さは必要なら mutation(Stryker)で点検する、と README に触れる程度でよい。
- モックを濫用しない。duration も expr も純関数なので出力ベースで検証する。
- 形式検証や TLA+ をリポジトリの git 管理下に置かない。痕跡を本体に残さない。
