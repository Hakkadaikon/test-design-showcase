# test-design-showcase

テスト設計のワークフローを、実際に動く TypeScript のコードで一直線に辿れる形にしたショーケースである。
振る舞いの抽出から手法の割り当て、テストへの橋渡しまでを、教科書どおりの順序で実地になぞれる。

題材は四つあり、いずれも縦に貫いて完成させてある。
最初の `duration`(時間文字列と秒数の相互変換)は、0段の振る舞い抽出から同値分割・境界値・性質ベーステスト・差分テストまでを通し、抽出台帳の各項目がテストに一対一で対応する。
残る三題材も同じ手順で、状態遷移テスト、構造網羅とメタモルフィック、非決定出力の階層化品質設計をそれぞれ実演する。

## ワークフロー

テスト設計を三段階で進める。
各段階の成果が次の段階の入力になり、最後にトレーサビリティ(振る舞いから手法、テスト名までの対応)が残る。

### 0段: 振る舞いを抽出する

いきなりテストを書かない。
まず、テストすべき振る舞いを洗い出す。

最初に曖昧な用語を定義する。
用語が未定義のまま進むと、境界と異常系の抽出漏れが最も多く生まれる。
`duration` であれば「受理する単位は `h`/`m`/`s` か」「単位の順序に制約はあるか」「ゼロ・負・空・桁あふれをどう扱うか」を、曖昧さが残らないところまで言語化する。

定義が固まったら、仕様の視点とコードの視点で独立に振る舞いを出し、両者の和を取る。
出した振る舞いを `T-001` から採番して台帳に落とす。
各振る舞いに種別(正常系・境界・異常系・非機能)を振り、正常系それぞれに「不正な入力なら? 境界なら? 並行に来たら?」と問うて、異常系を系統的に生やす。
原則は「過剰抽出は安全、漏れは危険」である。
挙げすぎた振る舞いは後で優先度を下げられるが、挙げ損ねた振る舞いはテストをすり抜ける。

### 手法割当

台帳の各振る舞いに、それを検証する手法を割り当てる。
同値分割、境界値、デシジョンテーブル、状態遷移テスト、性質ベーステスト、差分テスト、メタモルフィックテスト、モデルベーステスト、構造網羅(C0/C1/MC/DC)などから、振る舞いの性質に合うものを選ぶ。

### 検証へ橋渡し

台帳の T-ID がそのままテストリストになる。
TDD(Red、Green、Refactor)で一項目ずつ消し込み、振る舞いから手法、テスト名までの対応を保つ。

## 題材と手法の対応

| 題材 | 実演する手法 | 状態 |
|---|---|---|
| `duration`（`"1h30m"` と秒数の相互変換） | 同値分割、境界値、性質ベーステスト（往復で元に戻る不変条件）、差分テスト（独立な参照実装との突き合わせ）、ファジングと冪等性 | 縦に完成（33 tests green） |
| `order-fsm`（created→paid→shipped→delivered） | 状態遷移テスト（0-switch の全遷移と無効遷移）、モデルベーステスト、デシジョンテーブル | 縦に完成（26 tests green） |
| `expr`（`1+2*x` の評価） | 構造網羅（C0/C1/MC/DC）、デシジョンテーブル、メタモルフィックテスト（可換・結合） | 縦に完成（29 tests green） |
| `llm-validator`（非決定出力の階層化） | 非決定的出力の三層品質設計（入口で入力検証を網羅、中央で性質と複数サンプリングの合意、出口で後段バリデーション） | 縦に完成（19 tests green） |

## duration が縦の完成形である

`duration` は、ワークフローの全段階を実物で踏んだ題材である。

0段の抽出台帳は `tasks/` 配下にあり(git 管理外)、用語定義から `T-001` から `T-028` までの台帳、各振る舞いの種別までを実際にたどれる。
本実装は `src/duration/duration.ts` に置く。
差分テスト用の参照実装は `src/duration/reference.ts` に、本実装と同じ穴を共有しないよう独立に書いてある。
テストは `test/duration.test.ts` にあり、台帳の T-ID と一対一で対応する。

同値分割・境界値・性質ベーステスト・差分テストの四手法がすべて緑である。
往復で元に戻る不変条件(`parse(format(n))` が `n` に戻ること)は、広い範囲の入力を生成する性質ベーステストで叩いている。
各テストの `describe` 名と先頭コメントの T-ID をたどれば、台帳からテストまでの対応を追える。

## ディレクトリ構成

```
test-design-showcase/
├─ README.md            … 本ファイル（ワークフロー要約と題材・手法の対応）
├─ src/
│  ├─ duration/         … parse と format（縦に完成した題材）と README
│  ├─ order-fsm/        … 注文の状態機械（縦に完成した題材）と README
│  ├─ expr/             … 式評価器（縦に完成した題材）と README
│  └─ llm-validator/    … 非決定出力の階層化バリデータ（縦に完成した題材）と README
└─ test/                … 各 src に対応するテスト（手法ごとに describe を分ける）
```

各題材ディレクトリの README に、どの振る舞いになぜその手法を割り当てたかを一対一で書いてある。

## 使い方

```
npm install
npm test
```

`npm test` は vitest を一度だけ走らせる。
四題材あわせて 106 tests がすべて green である(duration 33・order-fsm 26・expr 29・llm-validator 19)。

カバレッジ率は目的にしない。
到達はテストの強さを保証しないので、強さを点検したいときは mutation(Stryker)を使う。
